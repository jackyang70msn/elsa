    │       │

    │       │                                       │
    │       │                                       ▼
    │       │                              排程觸發 → runCallback()
    │       │                                       │
    │       │                                       ├── clearSession()

    │       │                                       │     └── spawn claude CLI（bypassPermissions, maxTurns:25）

## 核心模組：Scheduler（`src/scheduler.ts`）

### 資料結構

```typescript
interface Schedule {
  id: string;           // "sched_{timestamp}_{random}" 格式
  botId: number;        // 所屬 worker bot ID
  chatId: number;       // 目標聊天 ID
  prompt: string;       // Claude 要執行的任務指令
  cronExpr: string;     // 5 欄位標準 cron 表達式
  humanLabel: string;   // 人類可讀標籤（如 "daily 9am"）
  createdAt: string;    // ISO 8601 建立時間
  lastRunAt: string | null;  // 最近一次執行時間
}
```

### ScheduleManager 類別

| 方法 | 說明 |
|------|------|
| `constructor(runCallback)` | 接收排程觸發時的回呼函式 |
| `start(schedules)` | 載入並啟動所有已儲存的排程 |
| `startTask(schedule)` | 驗證 cron 表達式 → 建立 `node-cron` task → 存入 `tasks` Map |
| `add(schedule)` | 持久化到 JSON + 啟動 cron task |
| `remove(scheduleId)` | 停止 cron task + 從 JSON 移除 |
| `removeAllForBot(botId)` | 移除某 bot 所有排程（bot 被移除時觸發） |
| `getForBot(botId)` | 取得某 bot 的排程清單 |
| `getAll()` | 取得所有排程 |
| `stop()` | 停止所有 cron task（daemon 關機時） |

### parseScheduleWithClaude() — AI 解析引擎

**輸入**：使用者的自然語言描述，如 `"daily 9am run tests and fix any failures"`

**處理流程**：
1. 組裝 prompt，要求 Claude 回傳 JSON `{cronExpr, humanLabel, prompt}`
2. 呼叫 `claude` CLI（`execFile`），參數：
   - `--print`：僅輸出結果
   - `--output-format json`：JSON 格式輸出
   - `--model claude-haiku-4-5-20251001`：使用成本最低的 Haiku 模型
   - `--max-turns 1`：只需一次回合
   - `--dangerously-skip-permissions`：跳過權限檢查
3. 透過 `stdin` 傳入 prompt（避免 Windows 上的 shell 跳脫問題）
4. 解析回應：先解析 CLI JSON wrapper（`.result` 欄位），再用 regex 提取 JSON
5. 超時設定：60 秒
6. 環境變數：剝除 `CLAUDECODE` 避免子程序拒絕執行

**輸出**：`{ cronExpr: string, humanLabel: string, prompt: string } | null`

**重要**：`prompt` 欄位會被 Claude 重寫為「適合 AI agent 自主執行的清晰指令」，而非原始使用者輸入。

## 完整操作流程

### 1. 建立排程（`/schedule`）

```
使用者: /schedule daily 9am run tests
    │
    ▼
[1] 輸入驗證（空輸入顯示說明文字）
    │
    ▼
[2] ctx.reply("Parsing schedule...")
    │
    ▼
[3] parseScheduleWithClaude("daily 9am run tests")
    │   └── spawn claude CLI (Haiku) → 自然語言 → {cronExpr, humanLabel, prompt}
    │
    ▼
[4] 解析失敗 → 回覆錯誤訊息，結束
    │
[5] 解析成功 → 建立 pending 確認
    │   ├── 存入 pendingScheduleConfirm Map（key = chatId）
    │   ├── 設定 5 分鐘超時 timer
    │   └── 回覆確認 UI（InlineKeyboard: Confirm / Cancel）
    │
    ▼
[6a] 使用者按 Confirm：
    │   ├── 清除 timer
    │   ├── 產生 scheduleId（sched_{timestamp}_{random}）
    │   ├── scheduleManager.add(schedule) → 持久化 + 啟動 cron
    │   └── 更新訊息為「Schedule saved」
    │
[6b] 使用者按 Cancel：
    │   ├── 清除 timer
    │   └── 更新訊息為「Schedule cancelled.」
    │
[6c] 5 分鐘超時：
        ├── 從 Map 移除
        └── 發送「Schedule confirmation timed out」
```

### 2. 排程執行（Cron 觸發）

```
node-cron 觸發
    │
    ▼
[1] 更新 lastRunAt → 寫入 schedules.json
    │
    ▼
[2] runCallback(botId, chatId, prompt, scheduleId)
    │   （定義於 daemon.ts:184）
    │
    ▼
[3] 查找對應 worker（activeWorkers Map）
    │   └── 找不到 → 記錄錯誤，結束
    │
    ▼
[4] bridge.clearSession(chatId) → 強制新會話
    │
    ▼
[5] bridge.sendMessage(chatId, prompt, callbacks, "bypassPermissions", 25)
    │   └── spawn claude CLI 子程序
    │       ├── permissionMode: bypassPermissions（自動核准所有工具）
    │       ├── maxTurns: 25（限制回合數）
    │       └── 所有審核回呼自動通過：
    │           ├── onToolApproval → "allow"
    │           ├── onAskUser → {}（空回應）
    │           └── onPlanApproval → true
    │
    ▼
[6] 執行結果處理：
    ├── 成功 → onResult
    │   ├── claudeToTelegram() 格式轉換
    │   ├── splitMessage() 分割長訊息
    │   └── sendMessage() → "<b>Scheduled task done</b>\n\n{result}"
    │
    └── 失敗 → onError
        └── sendMessage() → "Scheduled task failed: {error}"
```

### 3. 列出排程（`/schedules`）

**Worker Bot**：`scheduleManager.getForBot(botId)` → 只顯示該 bot 的排程

顯示格式：
```
Scheduled tasks for {repoName}

[1] daily 9am
Run tests and fix any failures
Last run: 2026-03-23 09:00:00

[2] every monday 9am
Write changelog from last week's commits
Never run
```

**Manager Bot**：`loadSchedules()` → 顯示所有 bot 的排程，按 bot 分組

### 4. 移除排程（`/unschedule`）

```
/unschedule 2
    │
    ▼
[1] 取得 bot 排程清單
[2] 驗證編號（1-based → 0-based）
[3] scheduleManager.remove(scheduleId)
    ├── task.stop()  → 停止 cron
    ├── tasks.delete() → 從記憶體移除
    └── saveSchedules() → 從 JSON 移除
[4] 回覆 "Removed: {humanLabel}"
```

## 6 大消費者分析

### 1. 資料庫（持久化儲存）

| 消費者 | 呼叫方式 | 資料內容 | 觸發條件 |
|--------|---------|---------|---------|
| `schedules.json` | `saveSchedules()` | Schedule 物件陣列 | 新增/移除排程、每次執行更新 lastRunAt |

**檔案位置**：`~/.elsa/schedules.json`
**權限**：`0o600`（僅 owner 可讀寫）
**格式**：JSON 陣列，每個元素為完整 Schedule 物件

### 2. 硬體/設備控制

**無**。排程功能不直接控制硬體。但排程觸發的 Claude 任務可能間接操作檔案系統。

### 3. 網路廣播

| 消費者 | 呼叫方式 | 資料內容 | 觸發條件 |
|--------|---------|---------|---------|
| Telegram API | `bot.api.sendMessage()` | 任務結果 / 錯誤訊息 | 排程執行完成時 |
| Claude CLI | `execFile('claude', ...)` | 自然語言解析請求 | 使用者建立排程時 |
| Claude CLI | `spawnClaude(...)` | 排程任務 prompt | 排程觸發執行時 |

### 4. 全域狀態變數

| 變數/Map | 型別 | 寫入者 | 讀取者 | 影響範圍 |
|---------|------|--------|--------|---------|
| `ScheduleManager.tasks` | `Map<string, ScheduledTask>` | `startTask()` / `remove()` / `stop()` | `remove()` | 控制 cron 排程的啟停 |
| `pendingScheduleConfirm` | `Map<number, {schedule, timer}>` | `/schedule` 指令 / callback handler | callback handler | 暫存待確認的排程 |
| `sessions` Map（ClaudeBridge） | `Map<number, string>` | `clearSession()` | `sendMessage()` | 排程執行時強制清除 session |

### 5. UI 更新

| UI 操作 | 呼叫方式 | 內容 | 觸發條件 |
|--------|---------|------|---------|
| 解析中提示 | `ctx.reply("Parsing schedule...")` | 進度回饋 | 使用者輸入排程 |
| 確認 UI | `ctx.reply()` + InlineKeyboard | Confirm/Cancel 按鈕 | 解析成功 |
| 儲存確認 | `ctx.editMessageText()` | Schedule saved + 詳情 | 使用者確認 |
| 取消提示 | `ctx.editMessageText()` | "Schedule cancelled." | 使用者取消 |
| 超時提示 | `sendMessage()` | "Schedule confirmation timed out" | 5 分鐘無回應 |
| 排程清單 | `ctx.reply()` | 編號清單 + lastRun | `/schedules` |
| 移除確認 | `ctx.reply()` | "Removed: {label}" | `/unschedule` |
| 執行結果 | `sendMessage()` | "Scheduled task done" + 結果 | 排程觸發完成 |
| 執行失敗 | `sendMessage()` | "Scheduled task failed: {err}" | 排程觸發失敗 |

### 6. 延遲/排程動作

| 延遲動作 | 方式 | 延遲時間 | 觸發條件 |
|---------|------|---------|---------|
| Cron 排程 | `node-cron.schedule(cronExpr, callback)` | 由 cron 表達式決定 | 排程確認後 |
| 確認超時 | `setTimeout(callback, 300000)` | 5 分鐘 | 排程解析成功後 |

## Daemon 生命週期中的排程管理

### 啟動順序

```
main()
  │
  ├── [1] 初始化 ScheduleManager（注入 runCallback）
  │        └── runCallback 綁定到 Telegram activeWorkers
  │
  ├── [2] 恢復 Telegram workers
  ├── [3] 恢復 Discord workers
  │
  ├── [4] scheduleManager.start(loadSchedules())
  │        └── 從 schedules.json 載入所有排程 → 啟動 cron tasks
  │
  └── [5] 啟動 health check timer
```

### 關機順序

```
shutdown()
  │
  ├── [1] clearInterval(healthCheckTimer)
  ├── [2] scheduleManager.stop()
  │        └── 停止所有 cron tasks + 清空 tasks Map
  │
  ├── [3] Telegram workers: abortAll → close tunnels → stop bot
  ├── [4] Discord workers: abortAll → close tunnels → destroy client
  └── [5] 刪除 PID 檔
```

### Worker 移除時

```
stopWorker(botId)
  │
  ├── bridge.abortAll()
  ├── scheduleManager.removeAllForBot(botId)
  │     └── 載入排程 → 篩選該 bot → 逐一 remove
  ├── tunnelManager.closeAll()
  └── bot.stop()
```

## 限制與注意事項

1. **runCallback 僅綁定 Telegram workers**：`daemon.ts:184` 的 runCallback 只從 `activeWorkers`（Telegram）查找 worker，Discord workers 不在此 Map 中。這代表**目前排程執行結果只能透過 Telegram bot 發送**。

2. **排程使用新 session**：每次排程執行前呼叫 `clearSession(chatId)`，確保不會繼承之前的對話上下文。

3. **自動核准所有操作**：排程執行時 `permissionMode: "bypassPermissions"`，且所有審核回呼都自動通過。這是因為排程為無人值守的自動執行。

4. **maxTurns 限制為 25**：防止排程任務無限執行。

5. **botId 跨平台問題**：Discord bot ID 是 string（snowflake），Telegram 是 number。`discord-manager.ts:379` 用 `String(s.botId)` 做比較。

6. **cron 表達式驗證**：`startTask()` 會驗證 cron 表達式，無效的會記錄錯誤但不會崩潰。

7. **無重試機制**：排程任務失敗後僅發送錯誤訊息，不會自動重試。
